═══════════════════════════════════════════════
CRITICAL REDESIGN OBJECTIVES
═══════════════════════════════════════════════

The current implementation has several major usability problems that must be solved during the redesign.

1. REMOVE GLOBAL DARK/LIGHT TOGGLE
- Remove storefront theme toggle completely.
- Users should never have to switch themes manually.
- Theme presets are controlled only by Admin Appearance Studio.
- Theme must remain consistent across all pages automatically.

2. FIX TEXT VISIBILITY ISSUES
- Current implementation causes white text on white backgrounds and dark text on dark backgrounds.
- Implement automatic contrast protection across all pages.
- Every page must remain readable regardless of selected theme preset.
- Admin cannot save invalid theme combinations.

3. REBUILD APPEARANCE STUDIO
Current appearance studio is difficult to use.

Replace JSON editing with visual controls.

Required:

- Upload image button
- Image preview
- Drag and drop uploads
- Replace image button
- Remove image button
- Live preview
- No JSON editing for banners
- No JSON editing for promo cards
- No JSON editing for category images

Everything must be editable visually.

4. MEDIA STORAGE REQUIREMENT
All uploaded images must:

- Upload to backend storage
- Save file metadata in database
- Be selectable from media library
- Be reusable throughout the website
- Never require manual path entry

5. SAVE FUNCTION RELIABILITY
Appearance Studio currently fails to save changes reliably.

Requirements:

- Save button always shows success/failure state
- Validation errors displayed clearly
- Unsaved changes warning
- Auto-save optional
- Restore Last Saved works correctly
- Preview works before save

### 0.4 — Database Analysis
- Database type (MongoDB/PostgreSQL/MySQL/etc.)
- Schema for each collection/table

- Indexes in use
- Any data integrity risks

### 0.5 — API Route Map
Generate a full API map in this format:
```
METHOD  /route/path         → handler_function()     [Auth Required: Y/N]
```
Include every route. Flag any undocumented, duplicate, or broken routes.

### 0.6 — Authentication Analysis
- Auth strategy and library
- Protected vs public routes
- Session/token storage
- Admin vs customer roles
- Any security gaps

### 0.7 — Admin Panel Analysis
- Admin pages and their functions
- Data flows (how admin actions affect the database)
- Current UX problems in admin
- Any hardcoded values or manual hacks in admin

### 0.8 — Current Problem Report
Produce a structured problem list across these dimensions:

**UI Problems** — Visual inconsistencies, bad typography, poor color use, outdated aesthetics
**UX Problems** — Friction points, missing affordances, confusing flows, broken interactions
**Performance Problems** — Unoptimized images, render-blocking scripts, no lazy loading, heavy bundles
**Mobile Problems** — Breakpoint failures, touch target sizes, overflow issues
**Accessibility Problems** — Missing alt text, contrast failures, no keyboard navigation, unlabeled forms
**Code Quality Problems** — Inline styles, magic numbers, duplicated logic, dead code
**Admin Problems** — Manual workarounds, JSON editing, limited controls, poor feedback
**Architecture Problems** — Tight coupling, missing abstractions, scalability blockers

### 0.9 — Risk Assessment
Flag anything that could break during the redesign:
- Tightly coupled frontend/backend code
- Hardcoded values that appear in multiple files
- Any deprecated libraries that need replacement
- External dependencies (payment gateways, maps, etc.)

## ═══════════════════════════════════════════
## QUALITY STANDARDS
## ═══════════════════════════════════════════

Every line of code you produce must meet these standards:

**HTML:**
- Semantic elements (`<header>`, `<nav>`, `<main>`, `<section>`, `<article>`, `<footer>`)
- All images have meaningful `alt` attributes
- All form inputs have associated `<label>` elements
- ARIA labels on interactive elements that lack visible text labels
- Correct heading hierarchy (only one `<h1>` per page)

**CSS:**
- All values reference design tokens (no hardcoded hex values or px values except in the token file)
- BEM naming convention OR utility-first (decide and be consistent)
- Mobile-first media queries
- No `!important` except to override third-party library styles

**JavaScript:**
- No `var` — use `const` and `let`
- No inline event handlers in HTML (`onclick=""`)
- All async operations have error handling
- No `console.log` in production code
- Event listeners cleaned up when elements are removed

**Accessibility:**
- Keyboard navigation works for all interactive elements
- Focus indicators visible (never `outline: none` without a replacement)
- Color contrast ≥ 4.5:1 for normal text, ≥ 3:1 for large text
- Reduced motion respected
